# Code Style

## 1. 소개
이 문서는 프로젝트에서 사용하는 코드 스타일을 정의한다. (구글의 Java 스타일 기준으로 함)

다른 프로그래밍 스타일 가이드와 마찬가지로, 다루는 문제는 형식 지정의 미적인 문제뿐만 아니라 다른 유형의 규칙이나 
코딩 표준에도 적용됩니다. 그러나 이 문서는 주로 우리가 보편적으로 따르는 **엄격하고 빠른 규칙**에 중점을 두고 있으며(사람이든 도구이든) 
명확하게 시행할 수 없는 조언은 제공하지 않습니다.

### 1.1. 용어
1. 클래스 - Class, Enum, Interface 또는 Annotation(@interface) 유형을 의미하기 위해 포괄적으로 사용
2. 클래스 멤버 - 중첩된 클래스, 필드, 메서드 또는 생성자를 의미하기 위해 포괄적으로 사용 
3. 주석이라는 용어는 항상 구현 주석을 나타냅니다 . 우리는 "문서 주석"이라는 문구를 사용하지 않고 대신 "Javadoc"이라는 일반적인 용어를 사용합니다.

### 1.2. 가이드 노트
이 문서의 예제 코드는 비표준 입니다 . 즉, 예제는 Google 스타일이지만 코드를 표현하는 유일한 스타일리시한 방법을 보여주지는 않을 수 
있습니다. 예제에서 선택한 선택적 형식 지정은 규칙으로 적용되어서는 안 됩니다.

## 2. 소스 파일 기본 사항
### 2.1. 파일 이름
소스 파일 이름은 대문자로 시작하는 CamelCase 이름이어야 합니다. 파일 이름은 클래스 이름과 일치해야하며 확장자는 .java 여야 합니다.

### 2.2. 파일 인코딩: UTF-8
소스 파일은 UTF-8로 인코딩되어야 합니다.

### 2.3. 특수 문자
#### 2.3.1. 공백 문자
줄 변경(종료) 시퀀스를 제외하고 ASCII 가로 공백 문자 ( 0x20 )는 소스 파일의 어느 위치에나 나타나는 유일한 공백 문자입니다. 
이는 다음을 의미합니다.  
* 문자열 및 문자 리터럴의 다른 모든 공백 문자는 이스케이프됩니다.
* 탭 문자는 들여쓰기에 사용되지 않습니다.

## 3. 소스 파일 구조
소스 파일은 다음 순서로 구성됩니다.

1. 라이센스 또는 저작권 정보(있는 경우)  
2. package 정보   
3. import 정보   
4. 하나의 최상위 클래스  

각 구성 요소 간 정확히 하나의 빈 줄을 이용해 섹션을 구분합니다.

### 3.1. 라이센스 또는 저작권 정보
소스 파일의 첫 번째 주석은 파일의 라이센스 또는 저작권 정보를 포함해야 합니다.

### 3.2. package 선언
패키지 선언문은 길이 제한 설정에 따른 줄 바꿈되지 않는다. 패키지 이름은 모두 소문자이다. 
패키지 이름은 일반적으로 회사의 도메인 이름을 역순으로 사용한다.  

### 3.3. import 정보
import 선언문은 길이 제한 설정에 따른 줄 바꿈되지 않는다. 
import 문은 다음 순서로 그룹화된다.
* 모든 정적 import
* 모든 비정적 import

각 블록은 빈 줄 하나로 구분되고, 그룹 내에서 ASCII 순서로 정렬된다.

정적 중첩 클래스에는 정적 가져오기가 사용되지 않습니다. 일반 수입품으로 수입됩니다.

### 3.4. 클래스 선언

#### 3.4.1. 하나의 최상위 클래스
소스 파일에는 하나의 최상위 클래스만 포함되어야 합니다. 

### 3.4.2. 클래스 멤버 순서
클래스의 멤버, 생성자(초기화), 메서드에 대한 순서는 가독성에 큰 영향을 미칠 수 있습니다. 그러나 가독성을 위한 정답은 없습니다. 
각 클래스마다 다른 방식으로 구성(나열)될 수 있습니다. 

중요한 것은 각 클래스에 대해 '설명 가능한 논리적 순서를 사용한다'는 것입니다. 
예를 들어, 새 메서드는 습관적으로 클래스 끝에 추가되지 않습니다. 그렇게 하면 논리적 순서가 아닌 
"추가된 날짜별 연대순" 순서가 생성되기 때문입니다.

#### 3.4.2.1. Override 메서드
동일한 이름의 클래스 메서드는 그 사이에 다른 멤버가 없는 단일 연속 그룹에 나타납니다. 
여러 생성자(항상 동일한 이름을 가짐)에도 동일하게 적용됩니다. 이 규칙은 메서드 간에 static 또는 같은 수정자(private)가 다른 경우에도 적용됩니다.

## 4. 포맷(Format)
### 4.1. 중괄호( {} )
#### 4.1.1 선택적 중괄호 사용
본문이 비어 있거나 단일 문만 포함된 경우에도 중괄호 는 if, else, for, do 및 while 문과 함께 사용됩니다.
람다식의 중괄호와 같은 선택적 중괄호는 선택 사항으로 유지됩니다.

#### 4.1.2. 비어 있지 않은 블록 : K&R 스타일
* 아래에 설명된 예외를 제외하고, 중괄호의 시작(여는 중괄호) 앞에는 줄 바꿈이 없습니다.
* 중괄호 시작 뒤에는 줄 바꿈이 있습니다.
* 중괄호 끝(닫는 중괄호) 앞에는 줄 바꿈이 있습니다.
* 중괄호 끝 뒤에는 줄 바꿈이 있습니다. - 중괄호가 문을 종료하거나 메서드, 생성자 또는 명명된 클래스의 본문을 종료하는 경우에만 해당됩니다. - 
  예를 들어 else 또는 쉼표(,) 가 뒤에 오는 경우 줄 바꿈이 없습니다.

예외 : 세미콜론(;)으로 끝나는 단일 문을 허용하는 위치에서는 중괄호의 앞에 줄 바꿈이 옵니다. 이런 블록은 일반적으로 스위치 문 내부와 같이 
지역 변수의 범위를 제한하기 위해 도입됩니다.

예제
```java
return () -> {
  while (condition()) {
    method();
  }
};

return new MyClass() {
  @Override public void method() {
    if (condition()) {
      try {
        something();
      } catch (ProblemException e) {
        recover();
      }
    } else if (otherCondition()) {
      somethingElse();
    } else {
      lastThing();
    }
    {
      int x = foo();
      frob(x);
    }
  }
};
```

#### 4.1.3. 빈 블록  
빈 블록이나 블록형 구성은 K&R 스타일일 수 있습니다. 또는 다중 블록 문의 일부가 아닌 한 문자나 줄 바꿈 없이 열린 후 즉시 닫힐 수 있습니다.
```java
  // This is acceptable
  void doNothing() {}

  // This is equally acceptable
  void doNothingElse() {
  }
```

```java
// This is not acceptable: No concise empty blocks in a multi-block statement
  try {
    doSomething();
  } catch (Exception e) {}
```

### 4.2. 블록 들여쓰기 : +2 spaces
새로운 블록이나 블록과 유사한 구문이 열릴 때마다 들여쓰기가 두 칸(space)씩 늘어납니다. 블록이 끝나면 들여쓰기는 이전 들여쓰기 
수준으로 돌아갑니다. 들여쓰기 수준은 블록 전체의 코드와 주석 모두에 적용됩니다.

### 4.3. 한 줄에 하나의 문
각 명령문 뒤에는 줄바꿈이 옵니다.

### 4.4. 열 제한 : 100  
Java 코드의 열 제한은 100자입니다. "문자"는 모든 유니코드 코드 포인트를 의미합니다. 
아래에 명시된 경우를 제외하고 이 제한을 초과하는 모든 줄은 섹션 4.5 줄 바꿈 에 설명된 대로 줄 바꿈되어야 합니다 .

예외:
1. 열 제한을 준수할 수 없는 줄(예: Javadoc의 긴 URL 또는 JSNI 메서드 참조)
2. package, import 문 
3. 셸에 복사하여 붙여넣을 수 있는 주석의 명령줄
4. 매우 긴 식별자는 드물지만 요청되는 경우 열 제한을 초과하는 것이 허용됩니다. 이 경우 주변 코드에 대한 유효한 래핑은 google-java-format에 의해 생성됩니다 .

### 4.5 줄 바꿈
**용어** : 합법적으로 한 줄을 차지할 수 있는 코드가 여러 줄로 나누어지는 경우 이 활동을 줄바꿈 이라고 합니다.  
모든 상황에서 줄 바꿈하는 방법을 정확히 보여주는 포괄적이고 결정적인 공식은 없습니다. 동일한 코드 부분을 줄바꿈하는 
유효한 방법이 여러 가지 있는 경우가 많습니다.

> **참고**: 줄 바꿈을 하는 일반적인 이유는 열 제한을 초과하는 것을 방지하기 위한 것이지만 실제로는 열 제한 내에 맞는 
> 코드라도 작성자의 재량에 따라 줄 바꿈될 수 있습니다.

> 팁: 메서드나 지역 변수를 추출하면 줄바꿈을 하지 않고도 문제를 해결할 수 있습니다.

#### 4.5.1. 중단 위치  
줄 바꿈의 기본 지시어는 다음과 같습니다: **더 높은 구문 수준** 에서 중단하는 것을 선호합니다. 또한:
1. 비할당 연산자에서 줄이 끊어지면 기호 앞에 줄이 옵니다.   
  (이것은 C++ 및 JavaScript와 같은 다른 언어의 Google 스타일에서 사용되는 것과 동일한 방식이 아닙니다.)  
   * 이는 다음과 같은 "연산자 모양" 기호에도 적용됩니다.
     * 점 구분 기호(.)
     * 메소드 참조의 콜론 두 개(::)
     * 유형 바인딩의 &퍼샌드(<T extends Foo & Bar>)
     * 캐치 블록의 파이프( .catch (FooException | BarException e) )
2. 대입 연산자에서 줄이 끊어지면 일반적으로 기호 뒤에 줄이 오지만 어느 쪽이든 허용됩니다.
   * 이는 향상된 for("foreach") 문의 "할당 연산자와 같은" 콜론에도 적용됩니다.
3. 메서드 또는 생성자 이름은 (그 뒤에 오는 여는 괄호( )에 계속 붙어 있습니다.
4. 쉼표( ,)는 앞에 오는 토큰에 계속 붙어 있습니다.
5. 람다의 본문이 중괄호가 없는 단일 표현식으로 구성된 경우 화살표 바로 뒤에 줄바꿈이 올 수 있다는 점을 제외하고는   
  람다의 화살표 근처에서는 줄이 끊어지지 않습니다. 

예제:
```java
MyLambda<String, Long, Object> lambda =
    (String label, Long value, Object obj) -> {
        ...
    };

Predicate<String> predicate = str ->
    longExpressionInvolving(str);
```

#### 4.5.2. 줄 바꿈 시 들여쓰기  
줄 바꿈 시 첫 번째 줄(각 연속 줄) 이후의 각 줄은 원래 줄에서 최소한 +4만큼 들여쓰기됩니다.
연속된 줄이 여러 개인 경우 들여쓰기는 원하는 대로 +4 이상으로 다양할 수 있습니다. 
일반적으로 두 연속 행은 구문적으로 평행한 요소로 시작하는 경우에만 동일한 들여쓰기 수준을 사용합니다.

수평 정렬에 관한 섹션 4.6.3에서는 특정 토큰을 이전 줄과 정렬하기 위해 다양한 수의 공백을 사용하는 권장되지 않는 관행을 다룹니다.

### 4.6. 공백  
#### 4.6.1. 수직 공백  
항상 하나의 빈 줄이 있는 위치:  
1. 클래스의 연속 멤버 또는 초기화 사이 : 필드, 생성자, 메서드, 중첩 클래스, 정적 이니셜라이저 및 인스턴스 초기화.  
    * 예외: 두 개의 연속 필드 사이에 빈 줄(그들 사이에 다른 코드가 없음)은 선택 사항입니다.  
    이러한 빈 줄은 필요에 따라 필드의 논리적 그룹을 만드는 데 사용됩니다.  
    * 예외: 열거 형 상수 사이의 빈 줄은 섹션 4.8.1 에서 다룹니다.  
2. 이 문서의 다른 섹션(예: '3.소스 파일 구조' 및 '3.3. import 문')에서 설명된 위치에는 빈 줄이 있을 수 있습니다.  

예를 들어 코드를 논리적 하위 섹션으로 구성하기 위한 문 사이와 같이 가독성을 향상시키는 위치에는 
빈 줄 하나가 나타날 수도 있습니다. 첫 번째 멤버/이니셜라이저의 앞과 클래스의 마지막 멤버/이니셜라이저 뒤에 빈 줄을 두는 것은 권장하지도 권장되지도 않습니다.

예:
```java
class MyClass {

    private int myField;
    // 첫번째 멤버의 앞에는 빈 줄이 없습니다.
   
    // 마지막 멤버의 뒤에는 빈 줄이 없습니다.
    private int anotherField;
    
}
```
여러 개의 연속된 빈 줄이 허용되지만 필수(또는 권장)는 아닙니다.

#### 4.6.2. 수평 공백(스페이스)   
언어 또는 기타 스타일 규칙에서 요구하는 경우를 제외하고 리터럴, 주석 및 Javadoc을 제외하고 단일 ASCII 공백도 다음 위치에만 나타 납니다 .

> 코드 작성 시 스페이스 규칙은 IDE 에서 대부분 자동으로 처리됨.
 
1. if, for, catch 등 예약어와 뒤에 오는 시작 괄호 '(' 사이에는 공백이 있어야 한다.
2. else, catch 와 같이 끝 중괄호 '}' 다음에 오는 예약어는 끝 중괄호 '}' 와 예약어 사이에 공백이 있어야 한다.
3. 시작 중괄호 '{'의 공백 처리 2가지 예외:  
   * @SomeAnnotation({a, b}) (공백이 필요하지 않음)
   * String[][] x = {{"foo"}}; (공백이 필요하지 않음, 8번 내용 참고) 
4. 이항 또는 삼항 연산자의 양쪽에는 공백이 있어야 한다. 이는 다음과 같은 기호에도 적용됩니다. 
   * 결합형 유형의 ('&') 기호 : <T extends Foo & Bar>
   * 여러 유형의 예외 처리를 위한 ('|') 기호 : (FooException | BarException e)
   * "foreach" 문의 기호 (':')
   * 람다식의 화살표 기호 : (String str) -> str.length()
   그러나 
   * 콜론 2개를(::)를 사용하는 메소드 참조의 경우 공백이 없음.(Object::toString)  
   * 멤버 변수, 메서드 사용을 위한 (.) 기호에는 공백이 없음.(object.toString())
5. `,:;` 기호와 형변환 캐스팅의 끝 괄호(')') 다음에 공백이 있어야 한다.   
6. 주석의 시작을 알리는 ('//') 기호의 앞과 뒤에는 공백이 있어야 한다. 공백 여러개도 허용. 
7. 변수의 선언 시 유형과 변수 명 사이에는 공백이 있어야 한다. 
8. 배열의 초기화 시 중 괄호('{', '}') 내부의 공백은 선택 사항   
   `new int[] {5, 6}` 과 `int[] { 5, 6 }` 모두 허용. 
9. Between a type annotation and [] or ....

이 규칙은 줄의 시작이나 끝 부분에 추가 공간을 요구하거나 금지하는 것으로 해석되지 않습니다. 코드 내부에서 공백 처리만을 다룬다. 

#### 4.6.3. 수평 정렬 : never required  
> 용어 참고: 수평 정렬(예제 참고)은 특정 토큰이 이전 줄의 다른 특정 토큰 바로 아래에 표시되도록 하기 위해 코드에 가변 개수의 추가 공백을 추가하는 방식입니다.

이러한 관행은 허용되지만 Google 스타일에서는 결코 요구되지 않습니다. 이미 사용했던 곳에서는 수평 정렬을 유지할 필요도 없습니다 .

예제: 
```java
private int x;
private Color color;

private int     x;
private Color   color;
```

> 팁: 수평 정렬은 가독성에 도움이 되지만 향후 유지 관리에 문제가 됩니다. 단 한 줄만 터치하면 되는 향후 변경 사항을 고려해보세요. 
> 이 변경으로 인해 이전에 만족스러웠던 형식이 엉망이 될 수 있습니다. 그리고 가로 정렬을 위해 사용자(프로그램 작성 시)가 지속적으로 
> 관리해야 하는 문제가 발생합니다. 이로 인해 무의미한 작업이 발생함과 동시에 최악의 경우 버전 기록 정보의 손상과 검토자의 업무 
> 증가(병합 충돌)만 가중시킵니다.   

### 4.7. 그룹화 괄호 : 권한  
선택적 그룹화 괄호는 작성자와 검토자가 괄호가 없으면 코드가 잘못 해석될 가능성이 없고 코드를 더 쉽게 읽을 수 없다는 점에 동의하는 
경우에만 생략됩니다. 모든 독자가 전체 Java 연산자 우선 순위 테이블을 기억하고 있다고 가정하는 것은 합리적이지 않습니다.

### 4.8. 특정 구성  
#### 4.8.1. 열거형(Enum) 클래스  
열거형 상수 뒤에 오는 쉼표(',') 뒤에 줄 바꿈은 선택 사항입니다. 추가 빈 줄(일반적으로 한 줄)도 허용됩니다. 
```java
private enum Answer {
  YES {
    @Override public String toString() {
      return "yes";
    }
  },

  NO,
  MAYBE
}
```
메소드가 없고 상수에 대한 문서가 없는 enum 클래스는 배열 초기화처럼 형식화될 수 있습니다(배열 초기화에 대한 내용은 4.8.3.1 참조).
```java
private enum Suit { CLUBS, HEARTS, SPADES, DIAMONDS }
```
열거형 클래스는 클래스이므로 클래스 형식 지정에 대한 다른 모든 규칙이 적용됩니다.

#### 4.8.2. 변수 선언  
##### 4.8.2.1. 선언당 하나의 변수  
모든 변수 선언(필드 또는 로컬)은 하나의 변수만 선언합니다. 다음과 같은 선언 `int a, b;`은 사용되지 않습니다.

> 예외: for 문의 헤더에는 여러 변수 선언이 허용됩니다. 

##### 4.8.2.2. 필요할 때 선언 
지역 변수는 소속된 블록이나 블록과 유사한 구문의 시작 부분에서 습관적으로 선언하지 않습니다.   
대신, 지역 변수는 범위를 최소화하기 위해 (합당한 범위 내에서) 처음 사용되는 지점에 가깝게 선언합니다. 
지역 변수 선언에는 일반적으로 초기화 프로그램이 있거나 선언 직후에 초기화됩니다.

#### 4.8.3. 배열  
##### 4.8.3.1. 배열 초기화 : 블록형 초기화  
모든 배열 초기화 프로그램은 선택적으로 "블록형 구조"인 것처럼 형식화될 수 있습니다. 예를 들어 다음은 모두 유효합니다( 전체 목록은 아님 ).
```java
new int[] {
  0, 1, 2, 3
} 
        
new int[]{
  0,
  1,
  2,
  3,
}
        
new int[] {
  0, 1,
  2, 3
}
        
new int[]
  {0, 1, 2, 3}
```

##### 4.8.3.2. C 스타일 배열 선언 : 허용되지 않음
C 스타일 배열 선언은 허용되지 않습니다. "String[] args, not String args[]."

#### 4.8.4. switch 문  
> 용어 참고: 스위치 블록의 중괄호 안에는 하나 이상의 명령문 그룹이 있습니다. 각 명령문 그룹은 하나 이상의 스위치 레이블(또는)과 그 뒤에 
> 하나 이상의 명령문(또는 마지막 명령문 그룹의 경우 0개 이상의 명령문)으로 구성됩니다. case FOO: default:

##### 4.8.4.1. 들여쓰기  
다른 블록과 마찬가지로 스위치 블록의 내용은 +2 들여쓰기됩니다.
스위치 레이블 뒤에는 줄바꿈이 있고 들여쓰기 수준은 마치 블록이 열리는 것처럼 +2 증가합니다.   
다음 스위치 레이블은 마치 블록이 닫힌 것처럼 이전 들여쓰기 수준으로 돌아갑니다.  

##### 4.8.4.2. Fall-through: 주석 처리  
스위치 블록 내에서 각 명령문 그룹은 종료되거나(break, continue, return 또는 예외 발생) 실행이 다음 명령문 그룹으로 계속될 수 있음. 
또한, 명령문의 실행 실패에 따른 스위치 블록 종료가 있을 수 있으며, 이는 주석을 이용해 설명되어야 합니다.(설명이면 충분) 
일반적으로 ('// fall through') 이 특수 주석은 스위치 블록의 마지막 명령문 그룹에는 필요하지 않습니다.

```java
switch (input) {
  case 1:
  case 2:
    prepareOneOrTwo();
    // fall through
  case 3:
    handleOneTwoOrThree();
    break;
  default:
    handleLargeNumber(input);
}
```
`case 1:` 뒤에는 주석이 필요하지 않고 명령문 그룹 끝에서만 주석이 필요합니다.

##### 4.8.4.3. Default 레이블의 유무
각 스위치 문에는 `default`를 위한 코드가 없더라도 `default`문이 포함됩니다. 

**예외**: enum 유형의 가능한 모든 값을 포괄하는 명시적 사례가 포함된 경우 유형에 대한 스위치 문은 `default`문 그룹을 생략 할 수 있습니다.  
이를 통해 IDE 또는 기타 정적 분석 도구가 누락된 사례가 있는 경우 경고를 발행할 수 있습니다.  

#### 4.8.5. Annotations  
##### 4.8.5.1 Type-use annotations
Type-use annotation 은 데이터 유형 바로 앞에 작성합니다.
Type-use Annotation 은 meta-annotated('@Target(ElementType.TYPE_USE)')를 가지고 있는 Annotation 이다.

예제: 
```java
final @Nullable String name;

public @Nullable Person getPersonByName(String name);
```

##### 4.8.5.2. Class annotations
클래스에 적용되는 Annotation 은 문서(주석) 블록 바로 뒤에 작성하며, 한 줄에 하나씩 나열한다.(즉, 한 줄에 하나의 Annotation).   
Annotation 줄 바꿈은 들여쓰기 수준이 증가하지 않습니다.

```java
@Deprecated
@CheckReturnValue
public final class Frozzler { ... }
```

##### 4.8.5.3. Method and constructor annotations
메서드와 생성자 선언에 대한 Annotation 규칙은 이전 섹션과 동일합니다.  
예제:
```java
@Deprecated
@Override
public String getNameIfPresent() { ... }
```

**예외**: 매개변수가 없는 단일 Annotation 의 경우 코드와 함께 작성될 수 있습니다.  
예제: 
```java
@Override public int hashCode() { ... }
```

##### 4.8.5.4. Field annotations
필드에 사용하는 Annotation 은 문서 블록 바로 다음에 나열되며, 이 경우 한 줄에 여러 Annotation(파라미터화된 Annotation 포함)을 나열할 수 있습니다.
```java
@Partial @Mock DataLoader loader;
```

##### 4.8.5.5. 매개변수와 지역 변수 annotations
매개변수 또는 지역 변수에 대한 Annotation 의 경우 특정 규칙은 없습니다.(하지만 Type-use Annotation 은 규칙 적용)

#### 4.8.6. 주석 
이 섹션에서는 주석에 대한 부분만을 다룹니다. Javadoc은 '7. Javadoc' 에서 별도로 다루어집니다 .
줄 바꿈 앞에는 임의의 공백이 오고 그 뒤에 구현 주석이 올 수 있습니다. 이러한 주석은 해당 행을 공백이 아닌 것으로 렌더링합니다.

##### 4.8.6.1. 블록 주석 style
블록 주석은 주변 코드와 동일한 수준으로 들여쓰기합니다. 블록 주석은 `/* ... */`스타일이나 스타일 `// .... `이 있을 수 있습니다. 
주석`/* ... */`의 경우 후속 줄은 이전 줄의 `*` 와 정렬하고, `*` 로 시작되어야 합니다.

```java
/*
  * This is
  * okay.
*/

// And so
// is this.

/* Or you can
 * even do this. */
```
주석은 별표나 기타 문자로 그려진 상자 안에 작성하지 않습니다.  
```java
/* 
 *   * ~ * ~ * ~ * ~ * ~ * ~ * 
 *   |         line1         |
 *   |         line2         |
 *   * ~ * ~ * ~ * ~ * ~ * ~ *
 */
```
 
> Tip: 멀티라인 주석을 사용할 때 자동 포맷팅이나 줄 바꿈이 적용되길 원한다면 `/* ... */` 스타일을 사용하세요. 
> 대부분의 포맷터는 '//' 주석에 대해서 포맷팅과 줄바꿈을 처리하지 않습니다. 

#### 4.8.7. Modifiers
클래스 및 멤버 한정자는 Java 언어 사양에서 권장하는 순서대로 작성합니다. 
```java
public protected private abstract default static final transient volatile synchronized native strictfp
```

#### 4.8.8. 숫자 Numeric Literals
long-valued integer literals use an uppercase L suffix, never lowercase (to avoid confusion with the digit 1). For example, 3000000000L rather than 3000000000l.
`long` 유형의 숫자는 대문자 'L' 을 붙여서 사용한다. 소문자 'l' 은 숫자 1과 혼동될 수 있음.(예: 3000000000L 과 3000000000l )

## 5. Naming
### 5.1. 모든 식별자의 공통 규칙 
식별자는 ASCII 문자와 숫자만 사용하며 다음에서 설명하는 소수의 경우에 밑줄을 사용합니다. 따라서 각 유효한 식별자 이름은 정규식`\w+`과 일치합니다.
Google 스타일에서는 특수 접두사 또는 접미사를 사용하지 않습니다. 예를 들어 kName, name_, mName 및 s_name는 Google 스타일이 아닙니다.

### 5.2. 식별자 타입 별 규칙  
#### 5.2.1. 패키지 이름 
패키지 이름은 소문자와 숫자를 이용해 작성합니다.
패키지 이름은 소문자와 숫자만 사용합니다(밑줄 없음). 연속된 단어는 단순히 함께 연결합니다.  
예: `com.example.deepSpace` 또는 `com.example.deep_space` 가 아닌 `com.example.deepspace`를 사용합니다.

#### 5.2.2. 클래스 이름  
클래스 이름은 대문자 시작 카멜케이스(UpperCamelCase)를 사용합니다.
클래스 이름은 일반적으로 명사 또는 명사 구문을 사용합니다. 예를 들어 Character 또는 ImmutableList. 
인터페이스 이름은 명사 또는 명사 구문(예: List)일 수도 있지만, 때로는 형용사 또는 형용사 구문(예: Readable)일 수도 있습니다.  

Annotation 타입에 대한 특정 규칙이나 잘 정립된 관례는 없습니다.

테스트 클래스는 Test 로 끝나는 이름을 가집니다. (예: `HashIntegrationTest` ) 
단일 클래스를 커버하는 경우 클래스 이름은 해당 클래스의 이름과 Test로 구성됩니다. (예: `HashImplTest` )  

#### 5.2.3. 메서드 이름  
메서드 이름은 소문자 시작 카멜케이스(lowerCamelCase)를 사용합니다. (예: `sendMessage` 또는 `stop`)

밑줄은 JUnit 테스트 메서드 이름에서 각 컴포넌트의 논리적 구성을 구분하기 위해 사용될 수 있습니다.( 예: transferMoney_deductsFromSource)  
테스트 방법의 이름을 지정하는 규칙은 없습니다.

#### 5.2.4. 상수 이름  
상수 이름은 대문자와 밑줄로 구성된 'UPPER_SNAKE_CASE'를 사용합니다.(예: `CONSTANT_NAME` ). 
정확히 상수란 무언인가요? 불변의 객체
단순히 객체를 절대 변경하지 않겠다는 '의지'만으로는 충분하지 않습니다.  

예제:
```java
// Constants
static final int NUMBER = 5;
static final ImmutableList<String> NAMES = ImmutableList.of("Ed", "Ann");
static final Map<String, Integer> AGES = ImmutableMap.of("Ed", 35, "Ann", 32);
static final Joiner COMMA_JOINER = Joiner.on(','); // because Joiner is immutable
static final SomeMutableType[] EMPTY_ARRAY = {};

// Not constants
static String nonFinal = "non-final";
final String nonStatic = "non-static";
static final Set<String> mutableCollection = new HashSet<String>();
static final ImmutableSet<SomeMutableType> mutableElements = ImmutableSet.of(mutable);
static final ImmutableMap<String, SomeMutableType> mutableValues =
ImmutableMap.of("Ed", mutableInstance, "Ann", mutableInstance2);
static final Logger logger = Logger.getLogger(MyClass.getName());
static final String[] nonEmptyArray = {"these", "can", "change"};
```
상수의 이름은 일반적으로 명사 또는 명사구입니다.

#### 5.2.5. 상수가 아닌 필드의 이름  
상수가 아닌 필드의 이름은 소문자 시작 카멜케이스(lowerCamelCase)를 사용하며, 일반적으로 명사 또는 명사구를 사용합니다.  
(예: `computedValues` 또는 `index`)

#### 5.2.6. 파라미터 이름  
파라미터 이름은 소문자 시작 카멜케이스(lowerCamelCase)를 사용합니다. (예: `inputStream`)
public 메소드에서는 한 문자로 된 매개변수 이름을 피해야 합니다.

#### 5.2.7. 지역 변수 이름  
지역 변수의 이름은 소문자 시작 카멜케이스(lowerCamelCase)를 사용합니다. (예: `index`)
불변의 속성을 가지고 있다고 하더라도 지역 변수는 상수로 간주되지 않으며, 상수로 스타일을 지정해서는 안됩니다.

#### 5.2.8. 변수 유형의 이름  
각 변수 유형 이름은 두 가지 스타일 중 하나로 이름이 지정됩니다.
* 단일 대문자 혹은 단일 숫자(opt). (예: E, T, X, T2)
* 클래스에 사용되는 형식의 이름(5.2.2. Class names)과 그 뒤에 대문자 T (예: RequestT, FooBarT).

### 5.3. 카멜케이스 : 정의 
"IPv6" 또는 "iOS"와 같은 약어나 특이한 구문이 있는 경우와 같이 영어 구문을 카멜 케이스로 변환하는 합리적인 방법이 
두 개 이상인 경우도 있습니다. 예측 가능성을 높이기 위해 Google 스타일은 다음과 같은 (거의) 결정론적 구성표를 지정합니다.

시작:
1. 구문을 일반 ASCII로 변환하고 어퍼스트로피를 제거합니다. (예: "Müller algorithm" -> "Muellers algorithm")
2. 이 결과를 공백과 나머지 구두점(일반적으로 하이픈)으로 단어를 나눕니다.  
    * 권장 사항: 단어가 이미 일반적으로 사용되는 카멜 케이스 모양을 갖고 있는 경우 이를 구성 요소로 분할합니다(예: "AdWords -> ad words")  
      "iOS"와 같은 단어는 실제로는 그 자체로 카멜 표기법이 아닙니다. 이는 어떠한 규칙에도 어긋나므로 이 권장 사항은 적용되지 않습니다.
3. 이제 모든 항목(약어 포함)을 소문자로 만들고 다음 항목의 첫 번째 문자만 대문자로 지정합니다.
    * ... 각 단어는 대문자 Camel Case를 생성 하거나
    * ... 첫 번째 단어를 제외한 각 단어는 소문자 카멜케이스를 생성합니다.
4. 마지막으로 모든 단어를 단일 식별자로 결합합니다.

원래 단어의 대소문자는 거의 완전히 무시됩니다. 

예:

|Prose form| Correct                           |Incorrect|
|---|-----------------------------------|---|
|"XML HTTP request"| XmlHttpRequest                    |XMLHTTPRequest|
|"new customer ID"| newCustomerId                     |newCustomerID|
|"inner stopwatch"| innerStopwatch                    |innerStopWatch|
|"supports IPv6 on iOS?"| supportsIpv6OnIos                 |supportsIPv6OnIOS|
|"YouTube importer"| YouTubeImporter, YoutubeImporter* | |

'YoutubeImporter' 의 경우 허용되지만 추천하지 않습니다. 

> 참고 : 일부 단어는 영어에서 모호하게 하이픈으로 연결됩니다. "nonempty" 및 "non-empty"는 모두 정확하므로 
> 메소드 이름 `checkNonempty` 및 `checkNonEmpty` 모두 가능합니다. 

## 6. 프로그래밍 실습  
### 6.1. @Override: 항상 사용 
메소드는 오버라이드되었을 때 마다 항상 @Override 를 사용합니다. 
여기에는 슈퍼클래스 메서드를 재정의하는 클래스 메서드, 인터페이스 메서드를 구현하는 클래스 메서드, 
슈퍼인터페이스 메서드를 다시 지정하는 인터페이스 메서드가 포함됩니다.

예외: 상위 메소드가 @Deprecated 인 경우 생략될 수 있습니다.

### 6.2. 알고 있는 예외사항 발생  : 무시하지 않는다. 
아래에 언급된 경우를 제외하고, 예외사항이 발생 시 아무 작업도 수행하지 않는 것은 올바른 일이 아닙니다.   
(일반적인 응답은 기록(로그)하는 것입니다. 또는 "불가능"하다고 간주되는 경우로 다시 에러(AssertionError)를 발생(전달)시킵니다.)

catch 블록에서 어떠한 조치도 취하지 않는 것이 정말로 적절한 경우 - 이것이 정당한 이유는 주석으로 설명

```java
try {
  int i = Integer.parseInt(response);
  return handleNumericResponse(i);
} catch (NumberFormatException ok) {
  // it's not numeric; that's fine, just continue
}
return handleTextResponse(response);
```
예외: 테스트에서 발견된 예외는 주석 없이 무시될 수 있습니다. 다음은 테스트 중인 코드가 예상 유형의 예외를 발생시키는지 확인하기 위한 
매우 일반적인 관용구이므로 여기서는 주석이 필요하지 않습니다.
```java
try { 
  emptyStack.pop();
  fail();
} catch (NoSuchElementException expected) {
}
```

### 6.3. 정적 멤버: 클래스를 사용  
정적 클래스 멤버에 대한 잠조를 정규화해야 하는 경우 해당 클래스 유형의 참조나 표현식이 아닌 해당 클래스 이름으로 정규화합니다.  
```java
Foo aFoo = ...;
Foo.aStaticMethod(); // good
aFoo.aStaticMethod(); // bad
somethingThatYieldsAFoo().aStaticMethod(); // very bad
```

### 6.4. Finalizers: 사용하지 않음  
`Object.finalize`의 재정의는 극히 드물다. 

> Tip: 하지마세요. 만일 당신이 그것을 꼭 해야 한다면, 우선 'Effective Java 의 8. "Avoid finalizers and cleaners"`을 
> 주의 깊게 읽고 이해한 다음 그리고 하지 마세요.

## 7. Javadoc
### 7.1. 포맷팅 : Formatting
#### 7.1.1. 일반적인 구성  
Javadoc 블록의 기본 형식은 다음 예제와 같습니다.
```java
/**
 * Multiple lines of Javadoc text are written here,
 * wrapped normally...
 */
  public int method(String p1) { ... }
```
  ... 혹은 한 줄 예제: 
```java
/** An especially short bit of Javadoc. */
```

기본 형식은 항상 허용됩니다. Javadoc 블록 전체(주석 표시 포함)가 한 줄에 들어갈 수 있는 경우 
한 줄 형식으로 대체될 수 있습니다. 이는 `@return` 과 같은 블록 태그가 없는 경우에만 적용됩니다.

#### 7.1.2. 단락 
하나의 빈 줄, 즉 정렬된 선행 별표('*')만 있는 줄은 단락 사이와 블록 태그 그룹(있는 경우) 앞에 나타납니다. 
각 단락은 `<p>`를 시작으로 다음 단어와 공백을 사용하지 않습니다. (`<p>My name`).   
`<ul>` 또는 `<table>`등의 다른 블록 수준 요소에 대한 HTML 태그 앞에는 `<p>`가 없습니다.

#### 7.1.3 블록 태그  
표준 "블록 태그"로 사용되는 @param, @return, @throws, @deprecated 는 앞에 작성된 순서대로 작성하며, 
네 가지 유형 모두 빈 설명과 함께 나타나지 않습니다.  
블록 태그가 한 줄에 맞지 않으면 연속 줄은 @의 위치에서 4칸(또는 그 이상)만큼 들여쓰기됩니다.


### 7.2. 요약 조각(Fragment)
각 Javadoc 블록은 간단한 요약 조각으로 시작됩니다. 이 조각은 매우 중요합니다. 이는 클래스 및 메서드 인덱스와 같은 
특정 컨텍스트에 나타나는 유일한 텍스트 부분입니다.  
이는 완전한 문장이 아닌 명사구 또는 동사구의 단편으로 구성된다. 요약 조각은 'A{@code Foo} is a....' 로 시작 하지도 않고, 
'This method returns...Save the record.' 처럼 완전한 명령문을 이루지도 않습니다.   
그러나 마치 완전한 문장인 것처럼 대문자로 표시되고 구두점이 표시됩니다.

> 팁: 일반적인 실수는 간단한 Javadoc을 작성하면서 `/** @return the customer ID */` 형식으로 작성하는 것입니다.
> 이는 잘못된 내용이므로로 `/** Returns the customer ID. */`와 같이 변경해야 합니다

### 7.3. Javadoc 사용 장소 
최소한 Javadoc은 아래에 설명 된 몇 가지 예외를 제외하고 모든 `public` 클래스와 
해당 클래스의 모든 `public` 또는 `protected` 멤버에 대해 존재합니다.  

'7.3.4. 비 필수 Javadoc' 에서 설명하는 것처럼 추가적인 Javadoc 내용이 존재할 수도 있습니다.

#### 7.3.1. 예외 : 설명이 필요한 멤버  
Javadoc은 `getFoo()`와 같이 설명하지 않아도 될 정도의 "간단하고 명백한" 멤버에 대해서는 선택 사항입니다.

> 중요: 일반적인 독자가 알아야 할 관련 정보를 생략하는 것을 정당화하기 위해 이 예외를 인용하는 것은 적절하지 않습니다. 
> 예를 들어, `getCanonicalName`라는 메소드의 경우 일반적인 독자가 "canonical name"이라는 용어가 무엇을 의미하는지 전혀 모르는 경우 
> Javadoc를 생략하거나 간략화하지 마십시오.

#### 7.3.2. 예외 : overrides 
상위 유형 메서드를 대체하는 메서드에 Javadoc이 항상 필요한 것은 아닙니다.
